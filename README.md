# MedTaskTracker – API трекера задач для медицинской информационной системы

## Описание

MedTaskTracker – это API-модуль для управления рабочими задачами персонала медицинской информационной системы (МИС).  
Он позволяет:

- Создавать, редактировать, удалять и просматривать задачи.
- Назначать теги задачам (системные теги защищены от изменения).
- Создавать периодические задачи с различными типами повторения (ежедневные, ежемесячные, по конкретным датам, чёт/нечет).
- Отслеживать индивидуальный статус для каждого дня выполнения (без дублирования записей на все даты).
- Переносить конкретное выполнение на другую дату (бонусная функция).

## Предположения и принятые решения

В процессе разработки мы сделали следующие разумные допущения:

1. **Одноразовая задача** – это задача с `recurrence_type: null`. Её дата выполнения хранится в поле `start_date`.
2. **Периодическая задача** – имеет заполненное поле `recurrence_type` (daily, monthly, specific_dates, even_odd) и обязательный `start_date` (первый день выполнения).
3. **Поле `end_date`** – необязательно. Если не указано, задача считается бесконечной (повторяется до тех пор, пока не закончится запрошенный диапазон).
4. **Генерация дат на лету** – мы не создаём записи на каждое повторение в базе данных. Вместо этого даты вычисляются в момент запроса в пределах переданного диапазона (`start_date` и `end_date` в запросе). Это решает проблему бесконечности и экономит место.
5. **Индивидуальный статус** – для хранения статусов отдельных экземпляров используется таблица `task_occurrences`. Если для конкретной даты нет записи, берётся статус из родительской задачи.
6. **Перенос выполнения** – реализован через поле `overridden_date` в `task_occurrences`. При его наличии дата экземпляра заменяется на указанную.
7. **Системные теги** – `отчетность`, `операции`, `звонок` создаются через `db:seed` и защищены от удаления и изменения имени.
8. **Фильтрация** – по умолчанию, если не переданы `start_date` и `end_date`, возвращаются задачи за текущий месяц.

## Локальный запуск (без Docker)

### Требования
- Ruby 3.4.10 (установлен через RubyInstaller, rbenv или rvm)
- PostgreSQL 16+ (запущен, известен пароль для пользователя `postgres`)

### Шаги

```bash
# Клонировать репозиторий
git clone <your-repo-url>
cd medical_task_tracker

# Установить зависимости
bundle install

# Настроить базу данных (создать файл .env с паролем)
# Создайте файл .env в корне проекта:
echo "DB_PASSWORD=ваш_пароль" >> .env

# Или отредактируйте config/database.yml, указав пароль в секциях development/test

# Создать и подготовить БД
rails db:create
rails db:migrate
rails db:seed   # создаст системные теги

# Запустить сервер
rails server
```

Приложение будет доступно по адресу `http://localhost:3000`.

## Запуск через Docker (рекомендуется)

### Требования
- Docker и docker-compose установлены и запущены.

### Шаги

```bash
# Клонировать репозиторий
git clone <your-repo-url>
cd medical_task_tracker

# Собрать и запустить контейнеры
docker-compose up --build
```

После успешного запуска API будет доступно по адресу `http://localhost:3000`.  
База данных PostgreSQL будет создана автоматически, миграции и seed-данные применены.

Чтобы остановить контейнеры, нажмите `Ctrl+C` или выполните:

```bash
docker-compose down
```

## API Эндпоинты

| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/api/v1/tasks` | Список задач с фильтрацией (`start_date`, `end_date`, `status`, `tag_id`) |
| GET | `/api/v1/tasks/:id` | Получение задачи с экземплярами в указанном диапазоне |
| POST | `/api/v1/tasks` | Создание задачи (параметры: `title`, `description`, `status`, `start_date`, `end_date`, `recurrence_type`, `recurrence_interval`, `recurrence_day_of_month`, `recurrence_dates[]`, `recurrence_even_odd`) |
| PATCH | `/api/v1/tasks/:id` | Обновление задачи |
| DELETE | `/api/v1/tasks/:id` | Удаление задачи (каскадно удаляются все связанные экземпляры) |
| POST | `/api/v1/tasks/:task_id/tags` | Добавить тег к задаче (body: `{ "tag_id": 1 }`) |
| DELETE | `/api/v1/tasks/:task_id/tags/:id` | Удалить тег у задачи |
| PATCH | `/api/v1/tasks/:task_id/occurrences/:date` | Обновить статус экземпляра на конкретную дату (body: `{ "status": "completed", "overridden_date": "2026-07-10" }`) |

**Статусы задач:** `pending`, `in_progress`, `completed`, `cancelled`.  
**Типы повторения:** `daily`, `monthly`, `specific_dates`, `even_odd`.

Подробное описание всех параметров и примеры ответов – в Swagger-документации (см. ниже).

## Дополнительные замечания

- Для работы с кириллицей в консольных тестах рекомендуется использовать **Postman** или встроенный **Rails console**, так как `curl` в Windows может искажать кодировку.
- Все ошибки валидации возвращаются с HTTP-статусом `422 Unprocessable Entity` и списком ошибок.
- При удалении задачи все связанные экземпляры (`task_occurrences`) также удаляются благодаря `dependent: :destroy`.
