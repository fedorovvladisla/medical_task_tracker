## 2. Выделение логики генерации дат в отдельный сервис

**Промт:**
> Логика генерации дат для разных типов повторения довольно объёмная. Где её разместить: в модели `Task` или вынести в отдельный сервис? Учитывай принцип единственной ответственности и тестируемость. Опиши плюсы и минусы каждого подхода.

**Ответ:**
- Рекомендовано создать отдельный класс-сервис `TaskRecurrenceGenerator`, который принимает задачу и диапазон дат, возвращает массив дат. Это упрощает тестирование, уменьшает размер модели и позволяет легко добавлять новые типы повторений.
- Модель `Task` будет содержать только метод-обёртку `occurrences_in_range`, который вызывает сервис и мержит с сохранёнными состояниями.
- Минус – дополнительный файл, но плюсы перевешивают.

**Решение:**  
Создан `app/services/task_recurrence_generator.rb` с методами для каждого типа повторения. Это решение обеспечило чистоту кода и лёгкость расширения.

## 4. Обработка невалидных дат (31 июня, некорректный формат)

**Промт:**
> Как правильно обрабатывать ситуации, когда пользователь передаёт несуществующую дату (например, 2026-06-31) в разных частях API: при создании задачи, в фильтрах, при обновлении статуса экземпляра? Нужно ли возвращать ошибку 400 или 422? Опиши единую стратегию.

**Ответ:**
- В модели `Task` поля `start_date`, `end_date`, `recurrence_dates` имеют тип `date`, поэтому при попытке сохранить невалидную строку ActiveRecord выбросит ошибку, и валидация не пройдёт. Это приведёт к `422 Unprocessable Entity`.
- В фильтрах (`TasksController#index`) мы используем `parse_date` с `rescue nil`, чтобы невалидные даты игнорировались и подставлялись значения по умолчанию. Это сделано для удобства клиента (не ломать запрос из-за опечатки).
- При обновлении статуса экземпляра (`OccurrencesController#update`) дата берётся из URL, и если она невалидна, `find_or_initialize_by` не сможет найти/создать запись, и сохранение вернёт ошибку валидации – клиент получит `422`.

**Решение:**  
Для фильтров – игнорируем невалидные даты и подставляем умолчания; для создания/обновления – возвращаем ошибку с деталями. Такое поведение считается разумным в контексте API.

## 5. Ограничение бесконечных задач при отсутствии `end_date`

**Промт:**
> Если пользователь создаёт задачу без `end_date`, она теоретически бесконечна. Как в методе генерации дат избежать бесконечного цикла и обеспечить производительность при больших диапазонах? Нужно ли искусственно ограничивать максимальный диапазон?

**Ответ:**
- Генерация всегда ограничена переданным диапазоном (`from_date`..`to_date`). В методе `TaskRecurrenceGenerator.generate` мы проверяем `current <= @to` и `end_date.nil? || current <= end_date`. Таким образом, даже без `end_date` генерация остановится на верхней границе диапазона.
- Для защиты от случайно огромных диапазонов (например, 100 лет) можно добавить проверку в контроллере, но в рамках ТЗ это не требуется, так как клиент обычно запрашивает месяц или неделю.
- Рекомендовано в документации указать, что для длительных периодов клиент должен передавать даты.

**Решение:**  
Ограничиваем диапазоном запроса – это решает проблему бесконечности без создания лишних записей. В документации указано, что для длительных периодов клиент должен передавать даты.

## 6. Диагностика и исправление ошибок при Docker-сборке (`secret_key_base` и PostgreSQL)

**Промт:**
> При попытке запуска `docker-compose up --build` возникли две ошибки:
> 1. PostgreSQL 18 не стартует из-за конфликта данных в томе.
> 2. Rails падает с `Missing secret_key_base for 'production' environment`.  
     > Как исправить без потери данных и с минимальными изменениями? Опиши варианты.

**Ответ:**
- Ошибка PostgreSQL связана с тем, что образ `postgres:18` использует новую структуру хранения, несовместимую со старыми данными. Решение: удалить том (или использовать `docker-compose down -v`) и сменить версию образа на `postgres:16` (стабильную и совместимую).
- Ошибка `secret_key_base` возникает, потому что в `Dockerfile` установлена `RAILS_ENV=production`, а для продакшена требуется ключ. Для разработки можно переключить окружение на `development` и добавить временный `SECRET_KEY_BASE`.
- В `docker-compose.yml` добавить переменные: `RAILS_ENV: development`, `SECRET_KEY_BASE: some_fake_key`.
- Если нужно сохранить production – использовать `rails credentials:edit` внутри контейнера.

**Решение:**  
Изменили `docker-compose.yml`, как рекомендовано, и запустили контейнеры. Это позволило успешно поднять сервер и протестировать API.

## 9. Бонусная задача: перенос выполнения на другую дату («отрыв» экземпляра)

**Промт:**
> В ТЗ есть бонус: врач может перенести задачу с одной даты на другую или отменить на конкретный день. Как это сделать без нарушения правила повторения? Предложи реализацию с учётом того, что задача может быть как одноразовой, так и периодической.

**Ответ:**
- Использовать поле `overridden_date` в таблице `task_occurrences`. При генерации дат, если для даты есть запись с `overridden_date`, то вместо исходной даты показывать перенесённую.
- Для отмены – установить статус `cancelled`.
- При обновлении через `PATCH /tasks/:task_id/occurrences/:date` клиент может передать `overridden_date` и новый статус.
- Это позволяет «оторвать» конкретный экземпляр от общего правила, не изменяя саму задачу.
